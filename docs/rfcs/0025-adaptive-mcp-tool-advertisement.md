# RFC-0025: Adaptive MCP tool advertisement

- Status: **Implemented** (2026-07-22)
- Depends on: 0016 (agent-grade MCP), 0008 (compliance pack), 0001 (decode registry)
- Blocks: nothing
- Origin: issue #137 (gap 3), reported from real agent sessions

## Abstract

`nuthatch mcp` advertises the same 12 tools to every nest, regardless of what the nest actually
indexes. Five of them (`balance`, `top_balances`, `flags`, `exposure`, `screen_status`) and the
`investigate-address` prompt are inert on a nest without a transfer-shaped decoder or without RFC-0008
compliance configured. An agent sees them in `tools/list`, calls one, gets `{"count":0}`, and either
concludes the index is empty or burns context rediscovering that the surface is partially inert. This
RFC makes advertisement a projection of what the nest *has*: the server exposes a derived capability
shape, and the bridge advertises only the tools that will return data.

## Motivation

The concrete failure mode: point an agent at a Uniswap-V3 pool nest (Swap events, no transfers, no
compliance) and ask "what are the top holders?". The agent sees `top_balances` advertised, calls it,
gets an empty result, and reports a false "the index is empty". RFC-0016's `schema` coverage section
mitigates but does not fix this: the menu (`tools/list`) and the coverage section become two sources of
truth that can disagree. The honest fix is to make the menu agree with the coverage by construction.

## Goals

- A nest advertises only the tools/prompts that will return data on it.
- The signal is *derived* from what the nest has (transfer-shaped decoder, compliance config), not an
  operator knob, so the surface cannot drift from reality.
- Zero new detection logic: reuse the exact signals the balance view and the compliance pack already
  use.
- Backward compatible: a bridge talking to an older nest (no shape endpoint) degrades to today's
  unconditional surface, never hiding a tool because a probe failed.

## Non-goals

- Changing `tools/call` dispatch: an explicitly-called tool still executes (and still returns empty on
  a shape it doesn't fit). This RFC governs *advertisement*, not access.
- Per-tenant or per-caller gating (that is the gateway's job, out of scope per non-negotiable on
  hosted multi-tenancy).

## Design

Two capability signals, both already computed elsewhere:

- **`transfers`** - the nest has an ERC-20/721 `Transfer`-shaped decoder (the same gate the IVM balance
  view is built from, `TableSchema::is_transfer_shaped`, mirroring `EventDecoder::transfer_columns`).
  Gates `balance`, `top_balances`.
- **`compliance`** - RFC-0008 compliance is configured: a threshold/velocity flag, or a `labels/` or
  `lists/` directory in the nest. Gates `flags`, `exposure`, `screen_status`, and the
  `investigate-address` prompt.

The running nest exposes this as `GET /shape` -> `{"transfers": bool, "compliance": bool}`. The
`nuthatch mcp` bridge fetches it on `tools/list` and `prompts/list` and filters the specs. The seven
generic tools (`status`, `schema`, `tables`, `table`, `sql`, `explain`, `entity`) are always
advertised.

The shape is a projection of live state, computed per request from the served `AppState`, so a
hot-swap upgrade (RFC-0020) that changes a nest's shape is reflected immediately, with no cache to
invalidate.

## Implementation

- `src/registry.rs`: `TableSchema::is_transfer_shaped()` - the schema-level mirror of
  `EventDecoder::transfer_columns()`.
- `src/serve.rs`: `GET /shape`, derived from `AppState` (`tables`, `threshold`, `velocity_threshold`,
  and the `labels/`/`lists/` dirs under the nest dir).
- `src/mcp.rs`: `Shape` + `fetch_shape()` (defaults to all-true on any probe failure); `tool_specs()`
  and `prompt_specs()` take `&Shape` and filter.

## Testing

- Unit: `tool_specs`/`prompt_specs` over each shape - bare nest (7 tools, no `investigate-address`),
  token nest (9 tools), compliance-configured (12 tools, `investigate-address` present).
- The existing `tools/list` handler test points at an unreachable base, so it exercises the
  fallback-to-all path (advertise everything on probe failure) - the safe default is tested.

## Risks

- **A misclassified nest hides a working tool.** Mitigated by the fail-open default (any probe failure
  advertises everything) and by reusing the exact signals the features themselves gate on, so
  advertisement and behaviour cannot disagree.
- **Filesystem probe in a request handler.** `/shape` is hit once per MCP session (on `tools/list`);
  two `is_dir()` checks are negligible.

## Alternatives considered

- **Pedro's `shape.toml`** (derived, content-addressed into the nest blob, regenerated by
  `nuthatch schema`): a stronger, pinned form. Deferred: the per-request projection is simpler, needs
  no new artifact, and reflects hot-swaps immediately. `shape.toml` remains a clean future step if the
  shape needs to travel with a published bundle.
- **Do nothing, lean on the `schema` coverage section**: preserves the two-sources-of-truth
  false-positive path (see Motivation).

## Open questions

- Should `investigate-address` be advertised on a transfers-only nest (where `balance` works but the
  compliance steps do not)? Currently gated on `compliance` only, since the prompt's distinguishing
  value is the compliance workflow. Revisit if usage suggests otherwise.
